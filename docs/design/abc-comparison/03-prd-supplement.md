## 1. 需求背景

### 1.1 业务现状

店长每日通过 PAD 端"日目标"页面进行加减货决策，当前评级列仅显示**本店产品 ABC**，决策依据单一。

### 1.2 面临问题

| 优先级 | 问题 | 具体描述 |
|---|---|---|
| P0 | 单店数据样本小 | 新店、淡季、偶发因素都会导致 ABC 失真 |
| P0 | 加货信心不足 | 店长不敢加货 → 缺货损失 |
| P1 | 减货依据弱 | 店长不敢减货 → 报损损失 |
| P1 | 标杆经验难传递 | 同类型门店的成功做法无法被复用 |

### 1.3 解决思路

在评级列显示 **本店 + 同类型门店组** 的双值 ABC，店长一眼可见自己与同类组的差距，提供"群体参照系"作为加减货决策依据。

## 2. 需求基本情况

| 要素 | 内容 |
| --- | --- |
| 需求提出人 | 运营管理部 + 区域经理 |
| 功能使用人 | 店长（PAD 端日目标编辑场景） |
| 受影响人 | 后台管理员（需配置对标组）、区域经理（看到对标后的目标）|
| 核心痛点 | 店长加货决策依据单一，缺乏同类店参照 |
| 需求价值 | 减少缺货 + 减少报损 + 培训标杆经验隐性传递 |

### 核心场景

**场景：店长编辑明日目标，遇到犹豫的产品**
- 人物：王店长，高新万达店（社区综合体店）
- 时间：每天 21:00 编辑次日目标
- 起因：手撕老面包本店评级 BC，目标销量不知道该填多少
- 经过：打开评级单元格 → 看到组内对标 → 发现社区综合体店组内 45 家店该产品平均 AA → 决定加大订货
- 结果：次日缺货风险降低，销售额提升

## 3. 业务规则与计算

### 3.1 评级偏离计算

```
偏离档数 = 组内ABC档位序号 - 本店ABC档位序号

档位映射：
AA = 1, AB = 2, BA = 3, BB = 4, BC = 5, CB = 6, CC = 7
（按销售额×完成度的二维矩阵展开）
```

| 偏离值 | 显示 | 颜色 | 业务含义 |
|---|---|---|---|
| 0 | `=` | 灰 #bfbfbf | 持平 |
| -1 | `↑` | 绿 #52c41a | 本店高 1 档 |
| ≤ -2 | `↑↑` | 深绿 #389e0d | 本店异常优秀 |
| +1 | `↓` | 橙 #fa8c16 | 本店低 1 档（建议加货）|
| ≥ +2 | `↓↓` | 红 #f5222d | 本店异常偏低（强建议加货）|

### 3.2 组内 ABC 计算（不剔除本店，决策 Q3）

```
组内某产品 ABC =
  对组内所有门店该产品的近 30 天销售额、完成度做 ABC 排名
  本店数据 包含 在内
```

**已知风险与应对：**
- 当组内门店数 < 5 时，组 ABC 易被本店主导，置信度低
- 应对：在抽屉中展示门店数量，并对 < 5 店的组加 "样本量较小" 标识

### 3.3 默认对标组选择规则

```
优先级：
1. 后台管理员手动配置的"门店级默认对标组"（决策 Q1）
2. 自动选择：在该门店所属、且"可作为对标组"开启的组中
   按 优先级 → 门店数升序 → 创建时间升序 选择
3. 兜底：所有门店
```

### 3.4 切换对标组生效范围（决策 Q4）

- 店长在前端列头切换对标组 → **仅当前会话生效**
- 关闭页面或刷新后回到默认对标组
- 不写入持久化存储

## 5. 数据模型

```mermaid
erDiagram
    STORE ||--o{ STORE_GROUP_REL : "属于多个"
    ANALYSIS_GROUP ||--o{ STORE_GROUP_REL : "包含多个门店"
    STORE ||--o| STORE_DEFAULT_GROUP : "门店级默认对标"
    ANALYSIS_GROUP ||--o| STORE_DEFAULT_GROUP : "默认对标组"
    ANALYSIS_GROUP ||--o{ GROUP_MATERIAL_ABC : "组内物料ABC"
    MATERIAL ||--o{ GROUP_MATERIAL_ABC : "物料"
    MATERIAL ||--o{ STORE_MATERIAL_ABC : "本店物料ABC"
    STORE ||--o{ STORE_MATERIAL_ABC : "门店"

    STORE {
        int id PK
        string name
        int store_type "店型"
    }
    ANALYSIS_GROUP {
        int id PK
        int tenant_id FK "租户隔离"
        string name
        string dimension "店型/等级/区域+品类等"
        boolean is_benchmark "是否可作为对标组(新增)"
        int priority "优先级(新增)"
    }
    STORE_GROUP_REL {
        int store_id FK
        int group_id FK
    }
    STORE_DEFAULT_GROUP {
        int store_id PK FK
        int default_group_id FK "门店级默认对标组(新增)"
    }
    GROUP_MATERIAL_ABC {
        int group_id FK
        int material_id FK
        date stat_date
        string abc_grade "组内ABC"
        int store_count "组内门店数"
    }
    STORE_MATERIAL_ABC {
        int store_id FK
        int material_id FK
        date stat_date
        string abc_grade "本店ABC"
    }
```

**核心新增表：**

| 表 | 说明 |
|---|---|
| `ANALYSIS_GROUP` | 已有表新增 `is_benchmark`、`priority` 两个字段 |
| `STORE_DEFAULT_GROUP` | 新增表，存储门店级默认对标组 |
| `GROUP_MATERIAL_ABC` | 新增表，T+1 计算每个组每个物料的 ABC |

## 6. 多租户考虑

| 考虑点 | 设计 |
|---|---|
| 数据隔离 | `ANALYSIS_GROUP.tenant_id` 字段，所有查询带租户过滤 |
| 组体系差异 | 每个租户的分析组维度可能完全不同，本设计不假设任何固定维度 |
| 配置入口 | 在租户后台，每个租户管理员独立配置自己的对标组 |
| 兜底规则 | 全局默认值"所有门店"组在每个租户内自动创建 |

---

## 7. 异常处理

| 异常场景 | 处理方案 |
|---|---|
| 门店不属于任何"可作为对标组"的组 | 评级列退化为单值显示（原版样式） |
| 组内门店数 = 1（仅本店）| 抽屉中标注"组内仅本店"，不显示对比箭头 |
| 组内门店数 < 5 | 抽屉中标注"样本量较小，仅供参考" |
| 后台管理员配置默认组后未保存 | 离开页面前提示"有未保存修改" |
| 当日组 ABC 数据未生成 | 显示 "—"，悬浮提示"数据计算中" |
